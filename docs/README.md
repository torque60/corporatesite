# corporatesite

## プロジェクト概要
このプロジェクトは、企業情報の提供、製品紹介、お問い合わせ窓口を備えたコーポレートサイトの構築と管理を目的としています。
一貫性のあるデザインと、機能ごとに整理されたディレクトリ構成により、長期的なメンテナンスとチームでの共同開発を容易にしています。

---

## 1. ディレクトリ構成
本プロジェクトは、機能ごとにディレクトリを分割するモジュール型の構成を採用しています。詳細は [file_structure.md](./file_structure.md) を参照してください。

```text
corporatesite/
├── index.html          # トップページ
├── about/              # 会社概要セクション
├── assets/
│   └── img/            # 共通画像リソース
├── contact/            # お問い合わせセクション
├── css/                # スタイルシート格納 (BEM準拠)
├── js/                 # JavaScriptファイル格納
├── locations/          # 拠点・アクセス情報セクション
├── products/           # 販売品目セクション
│   ├── detail/         # 製品個別詳細
│   ├── other-detail/   # その他販売品目
│   └── ...
│
└── docs/               # プロジェクトドキュメント (Gitコマンド、構成図等)
```

## 3. マルチバージョン管理 (ブランチ戦略)
本プロジェクトでは、用途や要件に合わせて3つのバリエーションを管理しています。`main` ブランチを基本とし、特定の機能変更を加えた長期存続ブランチを `variant/` プレフィックスで運用します。

| バリエーション | ブランチ名 | 概要 |
| :--- | :--- | :--- |
| **加工事例なし** | `variant/subtract-with-case-studies` | 基本的な構成のみを含むシンプルバージョンです。 |
| **加工事例あり** | `variant/with-case-studies` | 加工事例セクションを付加した、フル機能のバージョンです。 |
| **ニュース欄別パターン** | `variant/alternative-news` | ニュース欄のレイアウトや表示ロジックを独自に変更したバージョンです。 |

### バリエーション別開発フロー
1.  **基本修正・共通改善**: `main` ブランチで行い、各 `variant/` ブランチへマージして機能を波及させます。
2.  **機能の付加**: 加工事例などの特定機能は、それぞれの `variant/` ブランチで管理・開発します。

## 4. 開発ワークフロー (GitHub Flow)
本プロジェクトでは、**GitHub Flow** をベースとしたチーム開発を行っています。

### 3.1. 作業ブランチの作成
新しい機能追加やバグ修正を行う際は、必ず `main` ブランチから新しい作業ブランチを作成してください。
- 命名規則: `feature/機能名`, `fix/修正内容`, `docs/ドキュメント名`

### 3.2. コミットとプッシュ
変更を加えたら、意味のある単位でコミットし、GitHub上のリモートリポジトリへプッシュします。

### 3.3. プルリクエスト (Pull Request)
GitHub上でプルリクエストを作成し、他のメンバーにレビューを依頼します。
- 変更内容の説明を明確に記述してください。
- レビューが承認（Approve）されたら、`main` ブランチにマージします。

## 5. コーディング規約
品質を維持するため、以下の規約を遵守してください。

- **関心の分離 (Separation of Concerns)**:
  - HTMLには構造のみを記述し、インラインスタイルやインラインスクリプトを禁止します。
- **BEM (Block Element Modifier)**:
  - CSSクラス名は `Block__Element--Modifier` の形式を使用します。
- **DRY (Don't Repeat Yourself)**:
  - 同じスタイルやロジックの重複を避け、共通化を図ります。
